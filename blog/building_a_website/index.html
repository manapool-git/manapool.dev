Why does this website exist?

I’d always liked the idea of owning a personal website, but I’d never really found the motivation to learn how to build one.

The languages the web is built on never really clicked with me (nor did any kind of front-end development, to be totally honest), so I avoided it entirely. I’ve spent my career and all my personal projects manipulating data on the backend, far away from the UI and Node.js. I like to remain in the terminal as much as possible.

Over time though, it became an embarrassing blind spot for me. Family and friends would assume that because of my “fancy computer job”, I would be perfectly capable of building a simple website.

Well, I wasn’t. I had become a little one-dimensional and not general-purpose enough, so I decided to fix that by building this website from scratch.

I don’t care about your button animations, I just want to read the article

Computing power has increased exponentially, yet software seems to get slower and slower. Computers became so fast that developers no longer feel the need to optimise. The assumption among a large cohort of developers is that everyone must be using flagship phones or high-end PC hardware. Here’s a great rant about this that articulates it better than I ever could. Now, in fairness, software today does far more than it used to, but the trade-off often feels invisible to the user and some of the practises routinely used on the web are outragous.

I mean, is it really necessary to import 20 dependencies for a button? Or to have JS tracking on absolutely everything? Should a browser tab really need to eat 2GB of RAM?

Another problem that emerges when everyone is using the same frameworks and UI designs is that everything looks the same. In my opinion, the golden age of web design had to be the 2000s. Websites were really varied and oozed personality, I mean they were wildly inconsistent, but that was part of the charm.

To be fair, it hasn’t been made any easier to write better web apps and this is far from a web-only phenomenon ([desktop slop example]). Frameworks developed to cope with the quirks of the modern web became more complex and abstracted — not to mention the expectations of what a UI should be have also changed. Sometimes your website has to be big and bloated, but your recpie website isn’t one of them. I just wish there were more resistance to this trend in generally, complexity without a good reason for it. And that’s one of the main targets for this website — AVOID BLOAT.

My philosophy with this website moving forward

I think performance has to be a key factor. I want to make sure this site is lightweight and snappy, but still remains aesthetically pleasing1. That’s not to say this website is perfectly optimised, but I will strive to trim the fat as much as possible. Most of all, I want this website to have flavour, style, and personality.

I’m still figuring this out. But I’d rather build something small and intentional than something impressive and heavy.

Thanks for reading.

  1. If the ultimate goal was to be as fast and slimmed down as possible, this website would be plain HTML with no styling. ↩︎
